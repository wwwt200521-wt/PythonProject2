---

## SPEC 开发规划流程

本技能引导用户按照 SPEC 范式，**依次**生成 4 个结构化开发文档。每个文档生成后，**必须等待用户审阅确认**，才能继续下一个。

### 核心原则

- **渐进式生成**: 一次只生成一个文档，避免用户被大量内容淹没
- **用户确认门控**: 每个文档必须经用户审阅通过后才进入下一个
- **溯源性**: 4 个文档逐层细化，requirement 源自 spec，api 实现 requirement，test 验证 api
- **中文优先**: 文档内容以中文为主，代码/接口名可用英文

---

## 文档生成顺序

```
spec.md  →  requirement.md  →  api.md  →  test.md
(规划)      (需求)             (接口)      (测试)
```

**每个阶段必须遵循以下步骤：**
1. 根据用户的需求描述，生成当前阶段的文档内容
2. 将内容写入对应的 `.md` 文件
3. 向用户展示文档摘要（核心要点）
4. **等待用户反馈** — 用户可能要求修改，修改后重新展示
5. 用户确认"没有问题"后，**主动提示**进入下一阶段

---

## 阶段一：生成 spec.md

### 目标
将用户的需求转化为结构化的项目规划文档，定义项目的目标、范围、核心功能和架构概览。

### spec.md 模板结构

```markdown
# 项目规划文档 (Specification)

## 1. 项目概述
- 项目名称
- 项目简介（一段话描述项目是什么、解决什么问题）
- 目标用户

## 2. 项目目标
- 核心目标（1-2个）
- 次要目标（可选）

## 3. 功能范围
### 包含功能 (In Scope)
- 功能1：描述
- 功能2：描述
- ...

### 不包含功能 (Out of Scope)
- 明确不做的事情
- ...

## 4. 技术架构概览
- 技术栈选择及理由
- 系统架构图（文字描述即可）
- 核心模块划分

## 5. 关键设计决策
- 决策1：选择X而非Y，原因...
- 决策2：...

## 6. 约束与假设
- 技术约束
- 业务约束
- 关键假设

## 7. 风险与应对
- 风险1 → 应对措施
- 风险2 → 应对措施
```

### 生成前必须向用户提问
在生成 spec.md 之前，如果用户的原始需求描述中存在模糊之处，**主动向用户提问澄清**，至少覆盖：
- 项目的核心目标是什么？（一句话说清楚）
- 有没有明确不做的功能？（界定范围）
- 有没有技术栈偏好或限制？
- 目标用户是谁？使用场景是什么？

### 生成后确认
生成 spec.md 后，提示用户：
> "spec.md 已生成，请审阅。如有需要修改的地方，请告诉我。确认无误后，我们将继续生成 requirement.md。"

---

## 阶段二：生成 requirement.md

### 目标
将 spec.md 中的功能规划细化为具体的、可验证的功能需求和非功能需求。

### requirement.md 模板结构

```markdown
# 需求文档 (Requirements)

## 1. 功能需求 (Functional Requirements)

### FR-1: [功能名称]
- 描述：详细描述该功能的行为
- 输入：用户输入什么
- 输出：系统输出什么
- 前置条件：执行前需要满足什么条件
- 后置条件：执行后系统状态如何变化
- 验收标准：如何判断该功能实现正确

### FR-2: [功能名称]
...

## 2. 非功能需求 (Non-Functional Requirements)

### NFR-1: 性能
- 响应时间要求
- 并发处理能力
- ...

### NFR-2: 安全性
- 认证/授权要求
- 数据保护要求
- ...

### NFR-3: 可用性
- 错误处理要求
- 用户提示要求
- ...

## 3. 用户故事 (User Stories)
- 作为 [角色]，我想要 [功能]，以便 [价值]

## 4. 验收条件汇总
- 条件1
- 条件2
- ...
```

### 生成要求
- 每个功能需求必须可验证（有明确的验收标准）
- 非功能需求要具体、可度量
- 用户故事需覆盖所有核心场景

### 生成后确认
生成 requirement.md 后，提示用户：
> "requirement.md 已生成，请审阅。如有需要修改的地方，请告诉我。确认无误后，我们将继续生成 api.md。"

---

## 阶段三：生成 api.md

### 目标
基于 requirement.md 中的功能需求，设计每个函数/接口的详细规范，**包含伪代码**。

### api.md 模板结构

```markdown
# API 接口文档 (API Specification)

## 1. 技术环境
- 编程语言
- 运行环境
- 依赖库

## 2. 数据结构定义
- 核心数据类型和结构体

## 3. 接口/函数规范

### Function: [函数名]
- **用途**: 简要描述函数的作用
- **对应需求**: 实现 FR-X [需求名称]
- **输入参数**:
  - `param1` (type): 描述
  - `param2` (type): 描述
- **返回值**: (type) 描述
- **异常处理**: 可能抛出的异常及处理方式
- **伪代码**:
```
function 函数名(参数1, 参数2):
    // 步骤1: 验证输入
    if 参数无效:
        return 错误("参数无效")
    
    // 步骤2: 核心逻辑
    result = 执行核心操作(参数1, 参数2)
    
    // 步骤3: 后处理
    final_result = 格式化输出(result)
    
    // 步骤4: 返回结果
    return final_result
```

### Function: [下一个函数名]
...
```

### 生成要求
- 每个 function 必须有**完整伪代码**，逻辑清晰到可以直接翻译为实际代码
- 伪代码中用注释标注关键步骤
- 函数之间如果存在调用关系，需要说明调用链
- 如有模块/包划分，需要说明目录结构

### 生成后确认
生成 api.md 后，提示用户：
> "api.md 已生成，请审阅每个函数的伪代码是否清晰完整。如有需要修改的地方，请告诉我。确认无误后，我们将继续生成 test.md。"

---

## 阶段四：生成 test.md

### 目标
基于 requirement.md 的验收条件 和 api.md 的函数规范，设计测试计划。

### test.md 模板结构

```markdown
# 测试计划 (Test Plan)

## 1. 测试策略
- 测试层级：单元测试 / 集成测试 / 端到端测试
- 测试框架选择
- 测试覆盖率目标

## 2. 测试用例

### TC-1: [测试用例名称]
- **对应需求**: FR-X [需求名称]
- **测试目标**: 验证什么行为
- **前置条件**: 测试前需要准备什么
- **测试步骤**:
  1. 步骤1
  2. 步骤2
  3. ...
- **预期结果**: 期望看到什么输出/行为
- **测试数据**: 需要准备的测试数据

### TC-2: [下一个测试用例]
...

## 3. 边界与异常测试
- 空输入
- 非法输入
- 超长输入
- 并发/竞态条件（如适用）

## 4. 测试执行计划
- 执行顺序
- 依赖关系
- 预计耗时
```

### 生成要求
- 每个测试用例必须与 requirement.md 的验收标准对应
- 覆盖正常路径 + 边界条件 + 异常路径
- 测试步骤要具体到可以直接执行

### 生成后确认
生成 test.md 后，提示用户：
> "test.md 已生成，请审阅。确认无误后，4个文档已全部完成。"

## 完成后的下一步指引

4 个文档全部生成完毕并确认后，向用户输出以下指引：

```
SPEC 开发文档已全部生成完毕：
- spec.md       — 项目规划
- requirement.md — 需求定义
- api.md        — 接口规范（含伪代码）
- test.md       — 测试计划

下一步操作建议：
1. 将上述 4 个文件复制到你的 practice 目录中
2. 对 AI 开发者（TREA）使用以下提示词：
   "1. 阅读 requirement.md
    2. 理解 spec.md
    3. 按照 api.md 的要求进行开发
    4. 按照 test.md 的计划进行测试，如果有问题，修改后重新测试，直到成功"
```

---

## 注意事项

- **不跳过确认步骤**: 绝不能在用户未确认当前文档的情况下生成下一个文档
- **保持一致性**: 4 个文档之间必须保持术语一致、ID 对应（如 FR-1 在 requirement 和 test 中指向同一功能）
- **适应用户输入**: 如果用户的需求描述已经很详细，可以减少提问；如果模糊，必须多提问澄清
- **中文描述 + 英文代码**: 文档内容和描述用中文，代码/接口名/变量名用英文
- **文件生成位置**: 默认生成在用户指定的目录中，如果用户未指定，询问用户希望生成在哪个目录
